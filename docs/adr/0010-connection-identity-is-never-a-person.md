## Context（強制する問題）

ADR-0005 §9.2（LOG-0034）は「顧客所有BigQueryにはテナント別サービスアカウントで接続する」と決めた。これは**BigQuery接続型に固有の認証方式**として書かれており、次の2つの問いには答えていない。

オーナーからの問い（2026-07-24）:

1. **「BQやSnowflakeなど多種多様なDBに接続するSaaSとして、アクセス制御はどうあるべきか」**
2. **「アクセス制御は、誰が接続したかによってされるべきではないか」**

この2問は §9.2 を**ソース非依存の規則へ一般化**することを要求し、同時に §9.2 が明示していなかったギャップを露出させた。

### 強制する力

- **接続先の能力差が大きい**。行レベル制御を持つもの（Snowflake の row access / masking policy、Postgres の RLS、BigQuery の行アクセスポリシー）と、**持たないもの**（MySQL、CSV/Parquet 等のファイル群）が混在する。BigQuery の行ポリシーは IAM プリンシパルに束縛されるため、共有サービスアカウント下ではエンドユーザー単位に効かない。
- **エンドユーザーはウェアハウスに存在しない**。利用者はテナント（vendor）のSaaSユーザーであり、Google ID も Snowflake ユーザーも持たない。B2B2Cでは1テナントに数千〜数万人規模になりうる。
- **可視範囲がサーバ側で事前導出できることに②キャッシュが依存する**（ADR-0005 §4・§6、実測ヒット率99.89% / LOG-0031）。
- **剥奪の即時性は一律でなければならない**（原則D・§9.1）。

### 用語の分離（この問いの急所）

「誰」には2つの意味があり、混同がこの領域の事故の源になる。

| | 意味 |
|---|---|
| **認可主体** | 可視範囲を決める主体。「あなたが誰か」で見えるものが変わる |
| **接続主体** | データソースに資格情報を提示する主体。「何としてクエリが実行されるか」 |

問い2は**認可主体**の話としては完全に正しく、**接続主体**の話としては成立しない。

## Decision

### D1. 接続主体は決して人間ではない

データソースへの接続には**テナント単位の機械アイデンティティ**を用いる。その役割は**爆発半径の限定のみ**であり、可視範囲の決定には一切関与しない。（§9.2 の「顧客ごとに1SA」を、ソース種別を問わない規則に一般化する。）

### D2. 認可主体はエンドユーザー本人

可視範囲は、テナントが署名したトークンで表明されたエンドユーザーIDに対して、**自層の単一の点**で評価する。この評価はソース種別に依存しない。（ADR-0005 原則E-② の維持・明文化。）

### D3. セキュリティ実装の数をコネクタ数に比例させない

**新しいデータソースの追加は「コネクタ（データを取り出す手段）の追加」であって、「アクセス制御機構の追加」ではない。** 保証が接続先の種別によって変わってはならない。

### D4. プッシュダウンは最適化であり、制御ではない

認可述語をソース側のクエリに押し込むことは**性能最適化として行ってよい**。ただし**それを安全性の根拠にしてはならない**。押し込みが失敗・未対応でも、遅くなるだけで可視範囲は変わらない、という状態を保つ。

> 理由：方言ごとのSQL書き換えに安全性を賭けると、**パーサのバグが即座に情報漏洩**になる。接続先がN種類あればセキュリティコードの面積もN倍になり、D3に反する。

### D5. 顧客指定プリンシパルでの接続は「既定」ではなく「オプション」

以下が**すべて**揃うテナントに限り、顧客が指定したプリンシパルでの接続をテナント単位のオプションとして提供しうる。

- エンドユーザーが顧客のデータチーム／アナリストで、既にウェアハウスIDを持つ
- 母集団が小さい（数十〜数百）
- 顧客のガバナンス部門が「当社のウェアハウスのポリシーが正」と要求している

**代償を明記する**：この形態では可視範囲を自層で導出できないため、②結果キャッシュの共有を放棄する。

## なぜ「人が接続主体」だと壊れるか

Looker Studio 型の2モードは、**どちらも接続主体が人間**である。

| モード | 帰結 |
|---|---|
| 所有者資格情報 | **所有者の権限が全閲覧者に漏れる** |
| 閲覧者資格情報 | **全員にウェアハウスIDが必要**（運用不能） |

つまりジレンマの原因は「どちらの人間か」ではなく、**人間が接続主体であること自体**にある。接続主体を機械アイデンティティにすると、誤って継承されうる人間の特権が存在しなくなり、片方の角が消える。

補足として、埋め込み形態では**エンドユーザーは何にも接続していない**。データソースに接続したのは導入時の顧客管理者である。したがって「接続した人の権限」を根拠にすると、評価される主体が管理者になり、所有者資格情報の問題そのものに帰着する。

エンドユーザー単位で接続する案（問い2をD5以外の形で採る案）が多接続SaaSで成立しない理由：

1. **アイデンティティが N×M に爆発する**（接続先の数 × エンドユーザー数）。
2. **権限管理が顧客のDBの中に出ていく**。ロール変更が顧客データチームのDDL作業になり、管理UIも監査も分断される。
3. **接続プールと②キャッシュが死ぬ**（資格情報がユーザーごとに異なるため結果を共有できない）。
4. **能力の下限は変わらない**。行ポリシーを持たないソースでは依然として守れない。
5. **失効がN系統になる**（機構も伝播遅延も接続先ごとに異なる）。

## Consequences

### 既に満たしているもの

- **D2** は実装済み・実測済み。ゲートがJWTを検証 → コントロールプレーン（`PgControlPlaneReader`）がロール→`DataScope` を解決 → executor が適用。**同一の資格情報・同一SQLでもユーザーごとに結果が変わる**ことを実BigQueryで確認済み（LOG-0033、全店157,900 vs s1限定65,000）。対応付けを保持するのは自社コントロールプレーンであり、接続先ではない（LOG-0039）。
- **D1** は §9.2 で決定済み。ただし後述のとおり**未実装**。

### 変わるもの・要るもの

| 項目 | 現状 | D1–D4 が要求すること |
|---|---|---|
| 接続主体 | `AdcTokenProvider` が単一のADC IDを使う（開発者アカウント／単一SA） | `TenantBinding` が projectId と資格情報参照（`datasources.connection_ref`）を持ち、テナント別の機械IDで接続する |
| 接続先の種類 | 1種のみ（`type check (type in ('bigquery'))`） | D3は**2種目が出て初めて検証される**。それまでは主張であって実証ではない（ARC-005） |
| ②行スコープの制御点 | ASTバインディングが**唯一**の制御 | D4に照らすと不足。下記ギャップを参照 |

### 明るみに出たギャップ（このADRの実質的な発見）

ADR-0005 原則C-3（ソース層＝最後の砦）は、**①テナント分離にしか掛かっていない**。テナント別データセット／IAM／RLSは他社データへの到達を防ぐが、**同一テナント内の②行スコープ（店舗フィルタ）にはバックストップが存在しない**。

具体的には、`QueryPolicy` の表定義に `scopeColumn` を書き忘れた表は `bind.ts:118` で `rowScoped === false` となり、**エラーにならず全行を返す**。これは原則E（「②をRLSで実装しない」— 実行時カスタムロールをDDLで扱えないため）の帰結であり意図的な設計だが、**D4の観点では「SQL書き換えという最適化手段を、②の唯一の制御にしている」状態**にあたる。

したがって②にもソース非依存のバックストップが要る。候補（未決）:

- **(a) 成果物ベースの認可** — レポート × スコープ等価類ごとに実体化し、認可を「この主体はこの成果物を取得してよいか」に還元する。キャッシュ単位と認可単位が一致するため両者の食い違いというバグのクラスが消え、列制御も「成果物に入れない」だけで済む（ADR-0005 §6の列マスキング未実装問題にも効く）。探索的クエリは扱えないため、配信経路と編集経路の2階建てが前提。
- **(b) 実行後の結果セット検証** — 返却行のスコープ列を突合し、範囲外が1行でもあれば fail-closed。ソース非依存だが列が結果に含まれる必要がある。
- **(c) allowlist の `scopeColumn` 必須化** — 書き忘れを構造的に不能にする。安いが、防ぐのは「書き忘れ」だけでバインダ自体のバグには効かない。

どれを採るかは ADR-0005 §10 の未決項目に加え、デザインパートナーのスコープ実態を見て別ADRで決める。

## 検討した代替案

| 案 | 却下理由 |
|---|---|
| 各ソースのネイティブ機構に全面委譲 | D3違反（O(N)）。保証が接続先の種別で変わり、機構を持たないソースでは何も言えない。失効がN系統になる |
| エンドユーザー資格情報での接続を既定にする | §9.2で棄却済みの理由（(a)〜(e)）が多接続ではさらに悪化する。上記1〜5 |
| 全顧客で単一の資格情報 | ASTバインディングが唯一の防壁になり、原則C-3がこの形態で無効化される（§9.2で既述） |
| 成果物ベースを今すぐ全面採用 | 有望だが配信/編集の2階建てを前提とし、Phase 1の範囲を超える。上記(a)として未決に積む |
