## 【アーキ設計 v1】キャッシュ × 認可 — 具体設計

> 目的：**「速い（キャッシュ）」と「テナント/ロールで安全（認可）」を、実装ミスが即漏洩にならない構造で両立させる**。
> 対象読者：実装者。本書は §2 の設計原則が結論。以降は実装詳細と失敗モード。

---

## 1. 解くべき問題（一文）

BIの応答は**すべて認可依存**（テナント・ロールで中身が変わる）。素朴にキャッシュするとテナント越境で情報漏洩する。しかしキャッシュしないと速度と原価が死ぬ。**「認可を壊さずにキャッシュする」**方法を決める。

---

## 2. 設計原則（これが結論・最重要）

### 原則A：シェルとデータを分離する（Evidenceの使い方を変える）
Evidenceは通常「ビルド時にデータをHTMLへ焼き込む」。マルチテナントではこれをやめる。

- **シェル（テンプレート）**：レポートのレイアウト・チャート定義・JS/CSS。**テナント非依存**。`report_id + report_version` でキャッシュ。中身は機密でない。
- **データ（結果セット）**：チャートを埋めるクエリ結果。**テナント依存・機密**。認可付きAPIで別途取得し、`tenant + scope + query + data_version` でキャッシュ。

> この分離が全ての土台。データをシェルに焼き込まない限り、越境の面は「データAPI」だけに封じ込められる。

### 原則B：キャッシュキーは「認証済みコンテキストの純粋関数」
**キャッシュキーにクライアント入力を一切混ぜない。** `tenant_id`/`role` は必ず**サーバ側でセッションから解決した値**だけを使う。

- 帰結：ユーザーは**自分のテナントのキャッシュエントリしかアドレスできない**。別テナントのキーは計算上作れない → キャッシュヒット自体が構造的にスコープされる。
- これが「実装ミスが即漏洩にならない」安全性の核心。クライアントが `tenant_id` を渡せる設計にした瞬間、この保証は消える。

### 原則C：多層防御（キャッシュ層だけを信用しない）
1. **エッジ認可ゲート**：キー導出＋セッション検証。
2. **MCP層**：SQLへ `tenant_id` 強制注入（ASTレベル）。
3. **データソース層**：ソース側の行/データセット隔離を最後の砦にする。分析データがBigQueryなら**テナント別データセット** or **行アクセスポリシー**、管理データのPostgresなら **RLS**（機構はソースで異なるが、役割＝アプリ層のミスが即漏洩にならない保険、は同じ）。→ 原則D・§9。
4. **ペイロード自己検証**：キャッシュ値の中に `tenant_id` を埋め、読み出し時に `ctx.tenant_id` と一致を assert。キー導出バグを最後に捕まえる belt-and-suspenders。

### 原則D：データ層を「分析データ」と「管理データ」に分ける
1つのRDBに全部を兼務させない（初稿の `DB: Postgres+RLS` を分離）。

- **分析データ**（ダッシュボードが見せる業務データ＝NL→SQLの実行先／Evidenceの接続先）→ **BigQuery を第一級の対応ソース**とする。列指向でBIスキャンに適し、実スキーマスパイク（thelook 12/12）で実証済み、GCPネイティブ。顧客が持つ他ソースも同じ `datasource` 次元で扱う（§4キー・§3の allowed datasources と整合）。
- **管理データ**（tenants / users / roles / permissions / 監査ログ / テナント別接続設定）→ **Postgres（Neon等）＋RLS**。小さく整合性が要る関係データ。サーバーレス無料枠で持ち出しほぼ0。
- **分離する理由**：性質が真逆（大・スキャン重い・顧客所有もあり得る vs 小・トランザクション的・自社所有）。1RDB兼務は分析スキャンで劣化し、隔離機構も混線する。NoSQL不採用の判断もここ（NoSQL→RDB移行コストを甘く見ない／SQLコアとRLSを壊さない、LOG-0026）。

### 原則E：「テナント分離」と「ロール認可」を別物として設計する
アクセス制御には性質の違う2軸があり、担当を分ける。

- **① テナント分離（硬い境界）**：t_alpha が t_bravo のデータを一切見得ない。アプリ層の `tenant_id` 強制注入 ＋ データソース層の行/データセット隔離（原則C）。**これが「Lookerより安全」の唯一の根拠**。
- **② ロール認可（設定可能な製品機能）**：テナント内で「店長」がどのページ/データスコープを見られるか。**アプリ層（MCPゲートウェイ）のロジック＋管理データのロール/権限テーブル**で判定。§6の `scope_hash` はこの②をキャッシュに正規化したもの。
- **②をRLSで実装しない**：実行時に顧客管理者が作るカスタムロール（Phase 2）をDDLレベルのRLSでは扱えない。①はRLS/ソース行制御の適所、②はアプリ層の適所。混ぜない。

## 3. キャッシュの3層

| 層 | 中身 | キャッシュキー | 機密性 | TTL/失効 | 置き場所 |
|---|---|---|---|---|---|
| ①シェル | レポートのHTML/JS/チャート定義 | `report_id : report_version` | 低（テンプレのみ） | 長期・immutable | CDN Edge |
| ②結果 | クエリ結果セット（集計値） | `tenant_id : scope_hash : query_id : params_hash : data_version` | **高** | version失効＋短TTL backstop | Edge KV / Redis |
| ③認可コンテキスト | session→(tenant, roles, allowed reports/datasources) | `session_id` | 中 | **短TTL（例60s）＋失効イベント** | Edge KV（or 署名クレームに内包） |

- BIのチャートは基本「集計後の小さな結果」なので②はKVに収まるサイズが大半。巨大結果はオブジェクトストレージに退避＋KVは参照だけ、で対応。

---

## 4. キャッシュキーの具体式（②結果層）

```
key = "v1:" + tenant_id + ":" + scope_hash + ":" + query_id + ":" +
      sha256(params_normalized) + ":" + data_version

  tenant_id      … セッションから解決（クライアント入力厳禁）
  scope_hash     … ロール群を「データスコープ」に正規化したハッシュ（§6）
  query_id       … レポート定義内のクエリ識別子（report_versionに紐づく）
  params_hash    … フィルタ等のパラメータを正規化してハッシュ
  data_version   … (tenant, datasource) ごとの単調増加トークン（§5）
```

- 先頭 `v1:` とテナント接頭辞で**名前空間化** → スキーマ変更時の一括無効化、テナント退会時の一括パージが可能。
- キーは暗号学的ハッシュを含めて衝突を排除。

---

## 5. 失効（invalidation）— version token 方式

**TTL頼みにしない。キーにバージョンを含め、"immutable key" で自動ミスさせる。**

- `data_version`：`(tenant_id, datasource)` ごとの単調増加トークン。ETL実行・書き込みでインクリメント。
  - データが変われば `data_version` が変わる → キーが変わる → 自動的にキャッシュミス → 常に最新。**明示パージ不要**でstale配信が起きない。
- `report_version`：レポート定義（Markdown/SQL）が変わったら更新。**AIが編集した瞬間にbump** → シェル①と、当該レポートの②が自動で作り直される。
- 短TTLは「バージョン更新漏れ」への保険としてのみ置く（例：結果②は5〜15分）。

> 利点：明示パージのタイミングバグ（消し忘れ→古い値配信）を構造的に消せる。「消す」より「新しい鍵に移る」方が安全。

---

## 6. ロール爆発への対策：データスコープ等価類

ロールごとに別キーだと、ロールが増えるほどキャッシュが断片化しヒット率が落ちる。

- **ロールを「見えるデータの範囲（データスコープ）」に正規化**し、その等価類のハッシュ（`scope_hash`）をキーに使う。
- 例：「店長A」「店長B」がどちらも "自店舗のみ" スコープなら、**別ロールでも同一 scope_hash → キャッシュ共有**。
- ただし**行/列の見え方が実際に違うロールは必ず別 scope**（安全側に倒す）。「同じ結果を見るロールだけ」を束ねる。
- 列マスキングもスコープに含める（見える列集合が違えば別スコープ）。

## 7. リクエストの流れ（シーケンス）

```
エンドユーザー(埋め込みiframe/SDK)
   │  ①短命JWT（vendorのバックエンドが署名。tenant_id/user/rolesを含む）
   ▼
[エッジ認可ゲート]（常時稼働・薄い）
   │  1. JWT検証（署名・exp・aud）。不正なら即拒否
   │  2. 認可コンテキスト解決（③キャッシュ or クレーム展開）→ tenant_id, roles, allowed_reports
   │  3. 要求レポートが allowed_reports に含まれるか判定（含まれなければ403）
   │
   ├─(シェル要求)→ ①キャッシュ（report_id:report_version）→ ヒットで即返却
   │
   └─(データ要求)→ 4. サーバ側で key を導出（§4。クライアントのtenant_idは無視）
         │        5. ②キャッシュ参照
         │            ├ ヒット → ペイロード内 tenant_id == ctx.tenant_id を assert → 返却
         │            └ ミス  → 6. MCPへ（single-flightでstampede防止）
         ▼
      [MCP]  tenant_id をASTでSQLへ強制注入 → 検証
         │    → 分析データソースで実行(BigQuery等。テナント別データセット/行制御が最後の砦)
         │    → 監査ログ(管理データ=Postgres)
         │   結果に tenant_id を埋めて②へ保存（data_version付きキー）→ 返却
```

> ロール/権限・allowed_reports 等の**認可判定に使う管理データは Postgres（原則D）**、
> **クエリが集計する業務データは分析ソース（BigQuery等）**——ゲートは前者を読み、MCPは後者を叩く。

### 埋め込み認証のキモ（原則Bの入口）
`tenant_id` は**ブラウザではなく、顧客(vendor)のバックエンドが署名した短命JWT**から来る。ブラウザ経由の値は信用しない。これが崩れると全部崩れる。標準的な埋め込み分析の手法（signed JWT handoff）。

## 8. セキュリティ失敗モードと対策（レビュー観点）

| 失敗モード | どう漏れる/壊れるか | 対策 |
|---|---|---|
| クライアント供給の tenant_id | 別テナント指定で越境 | キーは**セッション解決値のみ**。クライアント値は完全無視 |
| キー衝突 | 別テナントの値を誤ヒット | 暗号学的ハッシュ＋`v1:`＋tenant接頭辞。ペイロード内tenant_id assert |
| 認可の陳腐化（ロール剥奪後もキャッシュ） | 剥奪済み権限で閲覧継続 | ③認可キャッシュは短TTL＋剥奪イベントで即失効 |
| データ陳腐化 | 古い数字を配信 | `data_version` をキーに内包（§5） |
| キャッシュスタンピード | 人気キーのミス集中でDB過負荷 | single-flight（同一キーの取得を1本化） |
| エラー応答のキャッシュ汚染 | 一時エラーを正解として配信 | 非200はキャッシュしない。負のキャッシュは超短TTLのみ |
| キー導出バグでの共有 | 実装ミスで越境 | ペイロードに tenant_id を埋め、読み出し時 assert（多層防御④） |
| 巨大結果でKV溢れ | 障害・切り捨て漏洩 | サイズ上限＋オブジェクトストレージ退避。切り捨て時はキャッシュしない |

## 9. 技術選定（v1の仮置き）

- **エッジ認可ゲート**：Cloudflare Workers / Vercel Edge（常時稼働の薄い層。これはv1で0円を諦める箇所）。
- **①シェル**：CDN（長TTL・immutable）。
- **②結果**：Edge KV（小集計）＋必要に応じRedis/オブジェクトストレージ。
- **③認可**：短TTL KV、または署名クレームに内包しゲートで展開。
- **分析データストア**（業務データ／NL→SQLの実行先／Evidence接続先）：**BigQuery**（列指向・サーバーレス・従量、無料枠で小規模は0円）。テナント分離は**テナント別データセット**を第一候補、または行アクセスポリシー/authorized view（§10で確定）。他ソース接続も同じ `datasource` 抽象で追加可能。
- **管理データ（コントロールプレーン）**：**Postgres（Neon等）＋RLS必須**。tenants/users/roles/permissions/監査ログ/接続設定。NoSQL不採用（原則D、LOG-0026）。
- **JWT**：vendorバックエンド署名の短命トークン（埋め込み）。社内ユーザー直利用はClerk/Supabase。

> Phase 1（デザインパートナー1社）では**上記を最小実装**：管理データPostgres＋分析ソース1経路（パートナーのBigQuery or テナント別データセット1つ）。多ソース対応コネクタ基盤は顧客が増えてから（過剰設計回避、[[requirements]] §1.5）。

### 9.1 認可データ（role→権限）も SoR は Postgres、速い読みは③キャッシュ

「カスタムロール→(アプリ, API権限)のマッピングはドキュメント型で読み多く強整合不要だからNoSQLで別管理してよいか」という論点への結論（LOG-0029）:

- **SoR（真実の置き場）＝ Postgres**（原則D）。理由:
  - **整合性要求は非対称**。付与の遅延は不便で済むが、**剥奪の遅延は「消したはずの権限が残る」＝セキュリティ事故**。RepChatは剥奪の即時性を `epoch＋デニーリスト` で保証しており、これは SoR が一貫している前提。結果整合のNoSQLをSoRにすると「いつ剥奪が効いたか」が曖昧になり、この保険とキャッシュ失効判断が噛み合わない。
  - **整合性**：role→permission は roles/tenants への参照。FK・enum/checkで孤児権限や権限名typoを弾ける（NoSQLは弾けない）。
  - **限界コストほぼ0**：既に動かすPostgresに小テーブル1つ。別ストアは運用増＋認可時のクロスストア結合＋整合性喪失を招き、対象はKB単位で水平スケールの利点も無い（過剰設計回避に逆行）。
- **速い読み＝既存の③認可キャッシュ**（短TTL KV / 署名クレーム）。read-heavyな認可判定はここで担保する。**「速いNoSQL読み」は②配信層として既に設計に在る**——SoRをNoSQLにする理由にはならない。SoRと配信層を混同しない。
- **書き味重視なら**：正規化テーブルの代わりに **Postgres の JSONB カラム**（例 `roles.permissions JSONB`）でドキュメント形状＋FK整合＋単一ストアを両立可。
- **将来**：ポリシーが多テナント×リソース単位ACLで複雑化したら **専用認可エンジン（OPA/Cerbos/SpiceDB）** を検討。今のスケールでは不要、その時に切り出す。

### 9.2 顧客所有ウェアハウス（接続型）の認証：テナント別サービスアカウント

「顧客のBQに繋ぐ場合、サービスアカウントで権限をもらうのか、OAuthで顧客ユーザー権限で実行するのか。SAだとテナント内のユーザーごとに違うアクセス制御が難しくならないか」への結論（LOG-0034）。

**前提の整理**：2つの別問題を分ける。①**クエリが誰として実行されるか**（ウェアハウスに提示するID）と、②**各エンドユーザーが何を見てよいか**（テナント内のユーザー単位認可）。

- **②はもともとウェアハウスのIDに委ねていない**（原則E）。ゲートが検証したJWT → コントロールプレーンのロール表 → `scope` → executorがASTで行フィルタをSQLに注入、が制御点。**同一SAで実行してもユーザーごとに結果が変わる**ことは実測済み（LOG-0033：同テナント・同SQLで全店 157,900 vs s1限定 65,000）。したがって「SAだとユーザー別制御が難しい」は成立しない。
- **OAuth（エンドユーザー権限で実行）は本製品形態には不適**：(a) 利用者はvendorのSaaSユーザーでGoogle IDを持たない、(b) 可視範囲がサーバ側で事前導出できず `scope_hash` による②キャッシュ共有が崩壊する（実測99.89%のヒット率はこれに依存、LOG-0031）、(c) iframe内で各ユーザーにOAuth同意とトークン更新を強いる、(d) 顧客が全エンドユーザーのBQ IAMを維持する羽目になり製品価値と逆行、(e) Phase 2のカスタムロールをIAMプリンシパルに対応付けられない。OAuthが適するのは別形態（アナリストが自社アカウントでログインするBIツール）。

**決定**：接続型では**顧客ごとに1つのサービスアカウント**を用い、顧客がそのSAに自分のデータセットへの `roles/bigquery.dataViewer` を付与する。認証情報の参照はコントロールプレーンにSecret Manager参照として保持（リポジトリには置かない、GR-001）。

- **粒度が要点**：全顧客で1SAにすると、その資格情報が全顧客のデータに到達可能になり、ASTバインディングが**唯一の**防壁になる（原則C-3がこの形態で無効化される）。**顧客ごとに1SAなら、ASTバインディングが完全に破綻してもトークンに権限がないため他社データに届かない** — ホスト型でRLS/データセット分離が果たす「最後の砦」の役割を、接続型ではIAMが果たす。
- **トレードオフ（正直に）**：SA方式では `SESSION_USER()` ベースの行アクセスポリシー等、**ウェアハウス側でユーザー単位の制御を効かせることはできない**（IDがSAに潰れるため）。規制業界の顧客が「BQ側でのユーザー単位監査」を要求した場合は再検討が要る。現状はアプリ層の制御がテスト済み・監査ログ付きであることで担保する。
- **コードへの含意**：`TenantBinding` が `dataset` に加え **projectId と資格情報参照** を持つ。ただしこれは `BindingResolver` の中身であり、executor の構造は既に「どのウェアハウスの、誰の資格情報で」を呼び出し時入力として受け取れる（LOG-0033）。**ホスト型/接続型は設定の違いであってコード変更ではない。**

## 10. まだ決めていない（要スパイクで確定）
1. **③認可を「短TTLキャッシュ」にするか「署名クレーム内包」にするか**：後者は失効が難しく前者はゲートのDB問い合わせが増える。トレードオフ実測で決める。
