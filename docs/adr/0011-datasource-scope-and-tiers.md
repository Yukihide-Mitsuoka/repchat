## Context（強制する問題）

オーナーからの問い（2026-07-24）:

1. Confluenceのような一見DBでないサイトに接続して統計ダッシュボードを作れたら便利では。ただし**その場でAPI実行するので実行コストが高い**印象。
2. GA4のようなデータ前提のツール、あるいはMAツールならもっと相性が良いか。
3. DWH／純粋なDB／GCS・S3上のjson・csv・parquetに限定すべきか。**ファイルはやめて完全にDWHとRDBのみ**にすべきか。

ADR-0010 は「どう繋ぐか」を決めたが「**何に繋ぐか**」を決めていない。D3（セキュリティ実装の数をコネクタ数に比例させない）は接続先が1種しかないため**主張であって実証ではない**（ARC-005）状態で残っている。

### 強制する力

- **D3**：保証が接続先の種別で変わってはならない。ソースのカテゴリを1つ増やすたび、この約束を再証明する面が増える。
- **D6・D7**：保守を発生させるのがベンダや顧客管理者で、ドリフトが開く側へ倒れる機構は採らない。
- **開発者1名・「小さく黒字」**（LOG-0021）。保守容量が天井。
- **コネクタ本数は既に商品化されている**（Fivetran / Airbyte 等）。1名で勝てる土俵ではなく、**この製品の差別化は埋め込みマルチテナントにおけるエンドユーザー単位の認可**（ADR-0010 D2）にある。
- **成果物ベース**（ADR-0010 の②独立層の唯一候補）に向かうなら、配信経路は内部的に既にファイルを読む系になる。

## Decision

### S1. ソースに要求する能力は4つだけ

| 要求 | なぜ |
|---|---|
| 資格情報のスコープを粗く絞れる | D1（テナント単位の機械ID）と D7 の粗い境界 |
| スキーマが安定している | レポートSQLが意味を保つため。変更日程をベンダが握るものは不可（D6） |
| 境界でデータを取り出せる | 認可は自層で行うので、ソース側のクエリ能力は必須ではない |
| **独自の細粒度ACLを持たない** | 持つ場合、無視すれば漏洩、ミラーすれば D7 違反（後述） |

4つ目が本ADRの実質的な選別条件である。

### S2. データソースを4階層に分ける

| 階層 | 対象 | 扱い |
|---|---|---|
| **1** | BigQuery | 実証済み（LOG-0033）。現行 |
| **2** | **オブジェクトストレージ（GCS / S3）** — Parquet を第一級、JSON/CSV は取り込み | **次に作る**（S3・S4） |
| **3** | Snowflake、Postgres/MySQL の**読み取りレプリカ** | パートナー要求で。※本番OLTP直結は顧客の運用リスクなのでレプリカ前提 |
| **4** | SaaSアプリのAPI（Confluence、Jira、Notion、GA4、MAツール等） | **自前のコネクタは作らない**（S5） |

### S3. 2つ目のソースは市場需要ではなく「D3を最も安く検証できるか」で選ぶ

D3 は現在ARC-005の意味で未実証である。2つ目のアダプタの第一の役割は**アーキテクチャの中心的な主張を試すこと**であり、市場カバレッジではない。

その基準ではオブジェクトストレージが最適である。**BigQueryから最も遠い**（サーバが無い、SQLエンジンが無い、IAM行ポリシーが無い）ため、最小の工数で最も強く試せる。Snowflakeを2番目にすると、BigQueryと似すぎていて**何も検証できない**。

副次的な利点として、ウェアハウスを持たないデザインパートナーを獲得できる（LOG-0021 の初期顧客像に合う）。

### S4. オブジェクトストレージは「直接クエリ対象」と「取り込み」に分ける

- **Parquet = 第一級のクエリ対象**。列プルーニング、行グループ統計、Hive形式のパーティションプルーニングが効くため「プッシュダウンが無い」は当たらない。**クエリ能力はソースに要求せず、こちらがエンジンを持ち込む**（DuckDB 等）。
- **JSON / CSV = 取り込み**。スキーマも型も統計も無く、**スキーマドリフトが静かにレポートを壊す**。到着時に境界で検証し（COD-010）Parquetへ正規化してから自前の管理下で提供する。ライブなクエリ対象にはしない。

アクセス制御の観点では、ファイルは**自前のACLモデルを持たない**ため、ミラーすべきものが存在しない。テナント別プレフィックス＋テナント別資格情報が粗い境界になり、D7 に適合する。

### S5. SaaSアプリのAPIコネクタは作らない

「顧客のパイプラインでウェアハウス、またはオブジェクトストレージに落としてもらい、それを読む」という位置づけにする。

## なぜConfluence型が落ちるのか — 理由はコストではない

コストは解ける。API即時実行が高いなら実体化すればよく、それは ADR-0010 が指した成果物ベースの方向そのものである。落ちるのは**アクセス制御**である。

Confluenceはスペース権限・ページ制限という**独自の細粒度ACL**を持つ。取り込んでダッシュボードにすると二択になる。

| | 帰結 |
|---|---|
| 元の権限を無視する | 読めないページの統計が見える = ADR-0010 が潰した Looker Studio 問題そのもの |
| 権限モデルを自層にミラーする | ページ単位ACLは任意かつ随時変わる。**D7 が「委ねてはいけない」とした細かく実行時に変わる境界を、逆向きに取り込む**ことになる。ミラーは静かに陳腐化し、**開く側に倒れる**（D6） |

安全な集計とそうでない集計を事前に区別できないことが問題を決定的にする。「先月の編集ページ数」は無害でも、「誰がどのページを」に一段ドリルダウンした瞬間に漏れる。

加えてスキーマ変更の日程をベンダが握る（D6）。

**着想自体は否定しない**：顧客が既にConfluenceをウェアハウスへ同期しているなら、そのダッシュボードは階層1の経路で今日でも作れる。実現できないのではなく、**コネクタを我々が作らないだけ**である。

## MAツールについて — 実地確認の結果

「Amplitudeがオブジェクトストレージ出力に対応しており、結果的にParquet形式でさえ出力すれば様々なMAツールのデータを取り込める」という見立てを確認した。**方向が逆だった**が、結論は変わらず、むしろ S4 を補強する。

- Amplitudeの**S3エクスポートはzipされたJSON**（時間単位でパーティション、1行1イベントJSON）である。
- **Parquetは取り込み（S3をソースとする）側**にコンバータ設定として現れる。

つまり **Parquetだけを受け付ける設計では、MAカテゴリは「ついてこない」**。ベンダがエクスポートで吐くのはJSON/CSVだからである。S4 の取り込み経路（境界で検証してParquet化）は衛生上の配慮ではなく、**このカテゴリを覆う実際の仕組み**である。

そして GA4 が示すより一般的な事実として、**データ志向のSaaSはほぼ全てウェアハウス連携を持つ**（Amplitudeも同様）。すなわち:

> データ志向のソースでは、**ウェアハウスが既に統合点になっている**。直結コネクタは、顧客のスタックが既にやっていることを、より下手に再実装することになる。

MAツールが Confluence 型よりマシなのは事実である（権限モデルが粗い、スキーマが比較的安定、データが元々イベント形）。しかし**マシであることと作るべきであることは別**であり、S5 は変わらない。

## Consequences

- `datasources.type` の check 制約が2値目を持つ（`bigquery` に加えオブジェクトストレージ）。ADR-0010 が「D3は2種目が出て初めて検証される」と書いた条件がここで満たされる。
- executor に2つ目のアダプタが要る。SQL方言と実行経路が変わる一方、**認可の実装は増えてはならない**（D3）。増えたなら設計が誤っている、という形で検証になる。
- LOG-0039 で「1つのデータソース形状しかないため投機的（COD-051）」として**注入**にとどめた `QueryPolicy` のテーブル許可リストは、2つ目の形状が現れる時点で**テナント別データになる**。
- 取り込み経路（JSON/CSV → 検証 → Parquet）は新しい運用対象である。D6 の観点では保守はパイプライン1本に集中し、失敗は**閉じる側**（データが更新されない／欠落が可視）に倒れる。
- 階層4を閉じることで、Fivetran/Airbyte 等と競合しない。顧客が既に持つパイプラインの**下流**に位置取りする。

## 検討した代替案

| 案 | 却下理由 |
|---|---|
| **完全にDWHとRDBのみ**（ファイルを切る） | ウェアハウスを持たないデザインパートナーを最初から失う（LOG-0021 の初期顧客像に反する）。成果物ベースに向かうなら内部的に既にファイルを読む系であり、切る理由が構造的に無い。Parquetのプルーニングにより性能上の反対理由も弱い |
| 2つ目のソースをSnowflakeにする | 市場的には自然だが BigQuery と似すぎており、**D3 を検証できない**（S3） |
| JSON/CSV もライブなクエリ対象にする | スキーマドリフトが静かにレポートを壊す。境界で検証しない入力を信頼しない（COD-010） |
| Confluence等のAPIコネクタを作る | 独自の細粒度ACLをミラーすることになり D6・D7 違反。コストではなくアクセス制御が理由 |
| MAツールの直結コネクタを作る | ウェアハウス連携が既に統合点として存在し、重複実装になる。ベンダがスキーマ変更日程を握る（D6） |
