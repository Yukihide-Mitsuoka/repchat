# 【要件定義書 v1】RepChat — マルチテナント型 AI-BI SaaS（Evidence × MCP）

> **v1の立ち位置**：v0（統合要件定義書）で見えた3つの楽観——①静的SSGとアクセス制御の矛盾、②サポート原価とCACの欠落、③工数の過小——を「設計判断」として潰したバージョン。数字は"守り"で引き直し、獲得とセキュリティ証明を工程に組み込む。**動くデモではなく、売れる最小プロダクトを定義する**ことを目的とする。

---

## 0. v0からの重要な方針転換（サマリ）

| 論点 | v0の前提 | v1の判断 | 理由 |
|---|---|---|---|
| インフラ | 深夜0円の完全サーバーレス静的配信 | **動的配信＋積極キャッシュ。小さな常時稼働ベースラインを許容** | 顧客の顧客に見せる埋め込み画面でコールドスタート3〜5秒はSLA失格。0円は魅力より足枷。 |
| アクセス制御 | 静的HTMLをCDN配信しつつRBAC | **認可ゲートを通した動的配信を正とする** | 静的HTMLに焼き込んだ数字は行/ページ単位で守れない。ここが最大の設計矛盾だった。 |
| AI課金 | 無料枠10回/人＋超過5円を毎回明示 | **基本料金にAIを実質使い放題として内包。異常利用のみ従量で防御** | 「誰でも一瞬」と「毎回5円」は体験が喧嘩する。課金明示はコンバージョンを下げる。 |
| スコープ | 全機能同時リリース | **Phase制。まず1つの尖った価値に絞る** | RBAC＋課金＋CI/CD＋AI同時は過大。 |
| 獲得/セキュリティ | 記載なし | **GTMとセキュリティ証明を工程に明記** | 「セキュア」は内製solo では最も証明しにくい売り文句。ここを外すと完成後に売れない。 |
| 工数 | 3.5ヶ月（6.5人月） | **MVPで4〜5ヶ月、商用品質はPhase2まで含め7〜9ヶ月** | production-grade多テナントは各要素が単独プロジェクト級。 |

## 1. プロダクトの一文定義（v1で尖らせる）

> **「自社SaaSに"顧客向け分析ダッシュボード"を、テナント間データ漏洩を物理的に不可能にした状態で、AI補助つきで最短で組み込めるSaaS」**

- **主戦場を1つに絞る**：埋め込み分析（Embedded Analytics）× マルチテナント安全性。
- AI生成は「主役」ではなく「立ち上げを速くする強力な補助輪」と位置づける（後述の精度リスクを織り込むため）。
- 「Tableauより安い／Lookerより安全」は**証明できる形にしてから**言う（§8）。

## 1.5 事業モデルの確定（2026-07-16 決定）

> **狙いは「小さく確実に黒字化」。急成長／VC型は狙わない。**
> デザインパートナー1社から始め、少数社を深く支え、兼業のまま**月数十万の純利益が出る副収入事業**に育てる。この確定が以降のスコープ・価格・獲得・工数の全判断基準になる。

この確定から自動的に決まること（discovery-log §8 の未決事項の解決）:

| 論点 | 確定 | 根拠 |
|---|---|---|
| 狙う勝ち筋（§8.3） | **小さく黒字型** | 兼業＝ランウェイ圧力ゼロ＋サポート応答性の天井。急成長は構造的に無理筋 |
| デザインパートナー（§8.4） | **開発と並行で1社を口説く**。これが唯一のGTM。深く支え→事例→紹介 | 少数を深く。広告・広範な営業はしない |
| 「4ヶ月」の扱い（§8.5） | **締切でなく工数**。暦の遅延を許容し、品質・セキュリティを日付で削らない | 小さく黒字は急ぐ理由がない |
| 社数の制約（§8.6） | **当面3〜5社で意図的に絞る**。深さ優先 | 兼業のサポート時間の天井を尊重。これは弱点でなく設計 |
| 収支の見方（§7） | 「20社で黒字」でなく**「3〜5社で黒字」を成立条件**に読み替える | 目的地が変わったため |

### スコープへの波及（小さく黒字で削れるもの）
3〜5社を**手動でオンボード**する前提なら、Phase 2 の自動化群は**大きく後ろ倒しできる**:
- **Stripeメーター課金・無料枠プールのリアルタイム計算** → 不要。**月額固定・請求書払い**（or 単純サブスク）で足りる。
- **セルフサーブのテナント作成・データソース接続UI** → 不要。**こちらが手で初期設定**する。
- **Discord風カスタムロール** → 不要。**管理／編集／閲覧の3段固定**で当面回す。

→ これらを削ると**初期スコープが実質軽くなる**。ただし**テナント分離（RLS＋AST注入＋キャッシュキー）とNL→SQLコアは規模に関係なく死守**（3〜5社でも各社に顧客の顧客がいるため）。

### モデル選定方針（2026-07-16 決定）
- **既定：Gemini 3.5 Flash（Vertex AI, ADC認証）**。安価・高速。
- **Opus 4.8 は最終手段のみ**：プロンプト・リクエスト改善では精度が出ず、モデル変更以外に道がない時だけ検討。
- 根拠：2026-07-17のクリーン実測で **Gemini 3.5 Flash が NL→SQL 12/12 PASS・1問≒¥0.1**（[[nl2sql-accuracy-spike]] / issue #3、詳細は discovery-log §8.1/8.2）。→ 安いモデルで製品コアが成立し、LLM原価はほぼ無視できる。

## 2. スコープ（Phase制）

### Phase 1（MVP｜4〜5ヶ月）— "1社と本番で戦える"最小構成
- マルチテナント認証（1テナント＝1顧客企業）
- **MCPセキュリティゲートウェイ：全クエリに `tenant_id` 強制注入＋テナント分離をDB層でも二重化（RLS）**
- 動的ダッシュボード配信＋キャッシュ（Evidenceをレンダリングコアに、認可ゲート経由で配信）
- **シンプルな2〜3段ロール**（管理者／編集者／閲覧者）※Discord風のカスタムロールはPhase2へ延期
- AIレポート生成（NL→SQL→検証→描画）＋対話型問い返し
- 使用量メータリング（内部計測。顧客向け課金表示はまだ出さない or 内包）
- 監査ログ（誰が・いつ・どのデータに触れたか）※セキュリティ証明の土台

### Phase 2（商用強化｜+3〜4ヶ月）— "20社に配れる"
> **【§1.5「小さく黒字」確定により、この章は大幅に後ろ倒し】** 3〜5社を手動オンボードする間は下記の自動化は不要。**「顧客が増えて手動が限界になったら着手」**のトリガー式に変更。急いで作らない。
- Discord風カスタムロール（ノーコードでロール作成・ページ/データソース単位の制限）※3段固定で足りるうちは不要
- ~~Stripeメーター課金＋無料枠プールのリアルタイム計算・締め・返金整合~~ → **月額固定・請求書払いに変更**（メーター課金の機構自体を近期スコープから除外）
- 部分再ビルド／キャッシュ無効化の最適化（下記の技術負債の本命）※これは規模でなくデータ量で効くので、パートナーのデータ量次第で前倒しあり
- セルフサーブのテナント作成・データソース接続UI ※手動オンボードで代替できるうちは不要

### Phase 3（スケール）— 事例が出てから
- SOC2準備、pentest、SLA商品化、固定枠プランでコールドスタート撲滅

**やらないことの明示（v1）**：BYODB全種対応、リアルタイムストリーミング、モバイルアプリ、白ラベル完全カスタムは初期スコープ外。

## 3. アーキテクチャ（v1｜矛盾を解いた版）

```
[ 顧客の顧客（エンドユーザー / ブラウザ） ]
        │  埋め込みiframe or SDK
        ▼
[ 認可ゲート（常時稼働の軽量エッジ層） ]   ← ここが静的信仰の代わり
        │  ①セッション→テナントID/ロール解決
        │  ②キャッシュヒットなら即返却（爆速はキャッシュで担保、静的信仰は捨てる）
        ├───────────────► [ キャッシュ層 (CDN/Edge KV) ] 認可キー付きでテナント別に分離
        │  キャッシュミス時
        ▼
[ レンダリング層（Evidence コア） ]
        │  認可済みコンテキストでのみSQL実行
        ▼
[ MCP サーバー（セキュリティ＋AIオーケストレーション） ]
        ├── tenant_id 強制注入 / クエリ検証 / 監査ログ
        ├── NL→SQL 生成（LLM）→ 検証 → リトライ → 描画指示
        ├── 対話型問い返し（曖昧さはエラーでなく質問で返す）
        └── 使用量メータリング
        │
        ├── [ LLM API（Anthropic Claude / 上位モデル前提） ]
        ├── [ Stripe（課金・メーター） ]
        └── [ Postgres（Neon等）｜Row Level Security 有効 ]
                └─ アプリ層(MCP)の tenant_id 注入 ＋ DB層 RLS の二重防御
```

### 設計上の要点
- **爆速はキャッシュで担保する**（静的ビルドで担保しない）。キャッシュキーに `tenant_id + role + query_hash` を含め、テナント越境を構造的に不可能にする。
- **多層防御**：アプリ層（MCPのtenant_id注入）だけに頼らず、**データソース側の行/データセット隔離**で二重化。片方の実装ミスが即漏洩にならないようにする（これが「Lookerより安全」を主張する唯一の根拠になり得る）。
- **常時稼働は"薄いエッジ認可層"だけ**に限定。重い部分は従量。0円ではないが、顧客向けSLAを守れる最小コストに寄せる。

> **⚠️ データ層の最新設計は [ADR-0005](adr/0005-cache-and-authorization-architecture.md) を参照**（2026-07-17改訂）。上の図の `Postgres｜RLS` は初稿の単一DB前提。現行は **分析データ＝BigQuery（＋テナント別データセット/行制御）／管理データ＝Postgres＋RLS** の2分割、かつアクセス制御を「①テナント分離／②ロール認可」の2軸に分離（原則D・E）。
> 図解版（アーキテクチャ図・シーケンス図・ER図・DDLドラフト）: [system-design.md](system-design.md)

## 4. 機能要件（Phase 1）

### 4.1 認可ゲート（新設・v1の心臓部）
- 全リクエストでセッション→`tenant_id`/`role`を解決し、下流に**署名付きコンテキスト**として渡す。
- 認可の判定結果をキャッシュキーに含め、越境参照を物理的に遮断。

### 4.2 MCP セキュリティゲートウェイ
- LLM/ユーザー由来のSQLに対し、`WHERE tenant_id = :ctx.tenant_id` を**パーサレベルで強制注入**（文字列連結でなくASTで）。
- 危険操作（DDL/DELETE/UPDATE、他テナント参照の試み）をブロック。
- **全アクセスを監査ログに記録**（§8の証明資料の材料）。

### 4.3 AIレポートアシスタント
- NL→SQL→**自動検証（実行前のドライラン＋件数/型チェック）**→失敗時リトライ→描画。
- 曖昧な指示は**対話で問い返す**（例：「売上は受注日と確定日どちら基準？」）。
- **重要な運用前提**：AIが生成に失敗した場合の**フォールバック（テンプレート選択・手動編集導線・サポート導線）**を必ず用意。AIの精度を100%前提にしない。

### 4.4 ダッシュボード閲覧
- 認可ゲート経由の動的配信。初回以降はキャッシュで1秒以下を目標。

### 4.5 ロール（Phase1は簡易版）
- 管理者／編集者／閲覧者の3段。カスタムロールはPhase2。

## 5. 非機能要件（v1で明示）

| 項目 | 要件 |
|---|---|
| 可用性 | 顧客向け埋め込み画面は**コールドスタート起因の遅延を出さない**（薄い常時稼働層で担保）。目標p95 < 1.5s。 |
| セキュリティ | テナント分離をアプリ層＋DB層で二重化。全アクセス監査ログ。秘密情報の暗号化。 |
| 監査性 | 誰が/いつ/どのテナントの/どのデータに/何をしたかを追跡可能。 |
| 障害時のAI | LLM障害・生成失敗時もダッシュボード閲覧は継続（AIは付加機能として分離）。 |
| データ削除 | 顧客解約時のデータエクスポート／完全削除フロー（ロックイン懸念への回答）。 |

## 6. ビジネスモデル ＆ プライシング（v1）

### 6.1 課金思想の転換
- **基本料金にAI利用を実質内包**（"使い放題"と打ち出す）。
- 従量は「異常利用の防御弁」としてのみ設計し、通常利用では顧客に課金を意識させない。
- 毎回「5円かかります」は**廃止**（コンバージョンを下げるため）。上限に近づいた場合のみ管理者に通知。

### 6.2 プラン例（要検証・仮）
- **基本：3,000〜4,000円/ユーザー/月**（AI通常利用込み）。
- **公正利用上限**：組織あたり月 [人数×N回] を"通常利用の目安"とし、大幅超過時のみ追加プラン案内。
- 価格は§7の"守りの原価"で黒字が出る水準に後から合わせる（数字先行で決めない）。
- **【§1.5確定】課金方式は月額固定・請求書払い**（or 単純サブスク）。3〜5社にメーター課金＋Stripe手数料は過剰——**近期はStripeメーター機構を作らない**。デザインパートナーは無償/割引で入れ、事例化を優先。

## 7. 収支・原価計画（"守り"で再計算）

### 7.1 v0からの是正点
- **創業者人件費・CAC・Stripe手数料を0円計上しない**。
- LLMは**上位モデル前提**でコストを積む（精度＝製品価値のため）。
- サポート原価を現実化（コード指向＋AI生成の土台は問い合わせが多い）。

### 7.2 20社（200ユーザー）時の月次（保守的な仮置き｜要実測で更新）
| 費目 | v0 | v1（守り） | 備考 |
|---|---|---|---|
| 静的/エッジ配信 | 0円 | 5,000〜15,000円 | 薄い常時稼働層を許容 |
| ビルド/再生成 | 40,000円 | 30,000〜80,000円 | データ量・編集頻度で線形以上に増える前提 |
| 認証 | 0円 | 0〜10,000円 | 規模で有料化しうる |
| DB（Neon等） | 20,000円 | 30,000円 | RLS/クエリ増 |
| LLM | 5,000円 | **実測で〜数百円/月規模** | 【2026-07-17実測】既定=Gemini 3.5 Flash、1問≒¥0.1。当初の「上位モデル前提20〜50,000円」は桁違いに過大と判明。Opus 4.8は最終手段のみ |
| **サポート実工数** | 100,000円 | **要人月換算（実質フルタイム1人に迫る）** | v0最大の過小評価。ここを直視する |
| Stripe手数料 | 0円 | **売上×〜3.6%（≒21,600円）** | 計上漏れ |
| **創業者人件費** | 0円 | **機会費用として必ず計上** | "粗利72.5%"は自分の給料0円の数字だった |

> **結論**：「純利益435,000円/月・粗利72.5%」は**自分の給料と営業コストを0円と置いた粗利**であり事業利益ではない。v1では上記を入れて再計算し、**それでも黒字になる価格・原価構造**を成立条件とする。CACとサポートが乗ると、20社は"損益分岐の入口"に近い可能性が高い——ここを楽観しないことが最重要。

### 7.3 【§1.5確定】成立条件は「3〜5社で黒字」に読み替え
- 目的地が「小さく黒字」に確定したので、**損益分岐の基準は20社でなく3〜5社**。
- 兼業＝創業者人件費を事業から取らない前提なら、月次の現金原価は「エッジ配信＋ビルド＋DB＋LLM＋（固定なら少額の）決済」のみ。**インフラ単体の損益分岐は1〜2社**、3〜5社なら明確に黒字。
- Stripeメーター機構を作らないぶん**Stripe手数料は請求書払いで最小化 or ゼロ**。
- 未確定の唯一の穴は**製品LLM費用**（§9スパイクのクリーン実測で確定）。ただし小さく黒字なら1社あたりの貢献が厚く低volumeなので、多少高くても吸収余地は大きい。**価格確定はクリーン実測の後**（[[nl2sql-accuracy-spike]] / issue #3）。

## 8. Go-to-Market ＆ セキュリティ証明（v0で欠落｜v1の必須章）

### 8.1 獲得（"20社をどう獲るか")
- ターゲットは埋め込みたいITベンダー＝**戦略的意思決定の相手**。衝動買いではない。
- **まず1社の設計パートナー（デザインパートナー）を無償/割引で獲得**し、本番導入・事例化。
- 相手が必ず要求するものを先回りで用意：**セキュリティ説明資料・SLA定義・撤退時データ削除フロー・監査ログ**。
- 事例1社 → セキュリティ資料 → 横展開、の順。**完成 ≠ 売れる**を工程で認める。

### 8.2 セキュリティを"謳う"でなく"証明する"
- 「Lookerより安全」の根拠を**二重防御（アプリ＋DB RLS）＋全監査ログ＋テナント分離テスト**として文書化。
- Phase3でpentest・SOC2準備。売り文句を検証可能にしてから前面に出す。

## 9. 最優先の技術検証（コードより先にやる"スパイク")

**製品リスクの本丸はNL→SQLの精度。ここが不安定だと他の全部が砂上の楼閣。**

1. **NL→SQL精度スパイク**：実データ・実スキーマ数種で、生成成功率・検証で弾けた率・要問い返し率を実測。→ AI/UX/価格を設計する前提データにする。
2. **テナント分離の攻撃テスト**：越境参照を意図的に試み、アプリ層/DB層のどちらでも止まることを確認。
3. **キャッシュ×認可の整合**：認可キー付きキャッシュで越境が起きないことを検証。

これらが通ってからPhase1本開発に入る。

## 10. 工数・スケジュール（現実版）

```
[スパイク: 3〜4週] NL→SQL精度 / テナント分離 / キャッシュ×認可 の検証
[Phase1: 4〜5ヶ月] 認証・MCP GW・動的配信・簡易ロール・AI生成・監査ログ → デザインパートナー1社本番
[Phase2: +3〜4ヶ月] カスタムロール・Stripeメーター課金・部分再生成最適化・セルフサーブ
