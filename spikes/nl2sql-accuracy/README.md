# NL→SQL Accuracy Spike

Answers discovery-log §8.1 (NL→SQL accuracy — the product's core risk) and
§8.2 (real per-report LLM cost) with measurements instead of opinions.
Tracked by issue #3.

## Method

- **Synthetic multi-tenant retail schema** ([schema.sql](schema.sql)) with both
  `ordered_at` (受注) and `confirmed_at` (確定) to exercise the known
  date-basis ambiguity. 3 tenants, deterministic seed ([seed.py](seed.py), `random.Random(42)`).
- **12 Japanese BI questions** ([questions.json](questions.json)) across 5 categories:
  simple aggregation / joins / time semantics / **ambiguity detection**
  (should ask, not guess) / **tenant safety** (cross-tenant asks must be refused).
- **Two-implementation verification**: expected answers are computed in
  **pure Python, without SQL** ([expected.py](expected.py)). Generated SQL passes only
  if its result set matches the independent computation.
- **Guardrails in the evaluator** ([run_spike.py](run_spike.py)): read-only connection,
  single-SELECT check, mandatory tenant filter, other-tenant reference scan —
  a miniature of the product's MCP gateway checks.

## How to run

```sh
python3 seed.py                                   # build spike.db (deterministic)
python3 expected.py                               # print expected answers (no SQL involved)
python3 run_spike.py                              # clean run: calls Claude API, measures accuracy + tokens + cost
python3 run_spike.py --model claude-sonnet-5      # compare a cheaper model
python3 run_spike.py --answers results/pilot_answers.json   # offline re-evaluation
```

The clean run needs `pip install anthropic` and `ANTHROPIC_API_KEY` (or an
`ant auth login` profile). It writes a Markdown report to `results/` with
per-question verdicts and measured cost in JPY.

### Pilot caveats (read before citing the 12/12)

1. **Generator ≈ evaluator author.** The pilot's SQL was generated by the same
   model (in the same session) that authored the question set and business
   rules. The result-match check against the independent Python computation is
   genuine verification, but the *should-ask/should-refuse* judgments and the
   prompt-familiarity advantage make 12/12 an **upper bound**, not the
   production number.
2. **Business rules were given, not inferred.** The prompt defines 売上/注文件数/
   直近30日/前月比 precisely. This mirrors the intended product design (rules
   live in tenant config), but it means the spike does *not* measure how the
   model behaves when rules are missing — that failure mode is Q8/Q9's
   clarify path, tested only lightly here.
3. **Cost not yet measured.** The pilot cannot record token usage. The clean
   run records real `usage` and computes JPY per question — run it before
   fixing pricing (discovery-log §8.2).

## What would falsify the optimistic reading

- Clean-run failures on Q3–Q7/Q12 (joins & time semantics) → NL→SQL needs a
  heavier verify/retry loop than budgeted.
- Q10/Q11 producing SQL instead of refusing → prompt-level tenant safety is
  insufficient; the AST-injection + RLS layers become load-bearing (they are
  designed to be, but the prompt should still refuse).
- Per-question cost ≫ ~¥5 → pricing assumptions in docs/requirements.md §7
  need rework.
